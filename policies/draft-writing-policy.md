# draft-writing-policy

10ページ大学の各ページ本文を制作するときの、draft執筆とレビューの運用方針。

各巻の作業ディレクトリでは、まず `abstract-0x.md` を書き、それを入力として `draft-0x.md` を書く。

`0x` はページ番号を2桁で表す。1ページ目なら `abstract-01.md` と `draft-01.md`、2ページ目なら `abstract-02.md` と `draft-02.md` とする。

## ファイルの役割

- `intro.md`：書き出し。ここは現在の形をキープする。
- `abstract-0x.md`：そのページの骨格を書く。
- `draft-0x.md`：`abstract-0x.md` を入力として、本文を書く。
- `review-0x.md`：`draft-0x.md` のセルフレビューまたは監修レビューを書く。
- `draft.md`：必要に応じて、`intro.md` と各 `draft-0x.md` を並べた組み上げ確認用として使う。

## abstract-0x.md に必ず書くこと

各ページの `abstract-0x.md` には、次の4点を先に書く。

1. 構成、節見出し、行数目安
2. 説明すること
3. 図示すること
4. 中身や仕組みがわかった楽しさがどこにあるか

この4点が固まってから、`draft-0x.md` に本文として反映する。

`abstract-0x.md` は、そのページを再執筆するための復元元である。レビュー後に重要な方針や不足が見つかった場合は、まず `abstract-0x.md` に戻す。

`abstract-0x.md` は、本文の言い回しより、説明順、節見出し、行数配分、図示内容、読者の理解の更新、小さな疑問と回収、読後に何がどう見え直すかを残す。再現性は、細かな文言指定より、ページの骨格と読後の見え方で担保する。

`abstract-0x.md` は、読者向け本文の短縮版ではなく、制作側の設計文書である。本文より長く、複雑で、高度な分析を含んでよい。

短く平易に見える本文の裏側で、概念の構造分析、比喩設計、誤解検出、失敗パターン、専門的背景、読者の認知変化を検討する。10ページ大学の品質は、表面の短さだけではなく、この設計密度によって作る。

この制作プロセスは、AI時代の解説書制作における重要な資産である。AIで効率よく制作しても、表面的な説明にせず、再現性ある高度な設計を経て本文を作ることを重視する。

節見出しは、説明のまとまりを読者へ示す案内板として計画する。各節の見出しは、本文で何を説明するかを短く示し、読者がいまどの段階を読んでいるかを見失いにくくする。

## 構造分析とシーン比喩

高度な概念、用語、仕組みを扱うページでは、`abstract-0x.md` に「構造分析とシーン比喩」の節を置く。

この節では、説明したい概念を次の順で分解する。

1. 概念・用語・仕組み: そのページで説明したい対象を列挙する。
2. 役割: 登場する部品、人、約束、処理の役割を書く。
3. 情報の流れ: 何がどこから来て、どこへ渡り、何に変わるかを書く。
4. 時間順の変化: 入力、変換、保存、読み戻し、表示、失敗などの順番を書く。
5. 失敗パターン: 約束のズレ、情報不足、読み違い、過剰な削減など、理解に効く失敗を拾う。
6. 類比できる日常シーン: 役割と流れが似ている常識的な場面を複数出す。
7. 採用するシーン比喩: 本文で使う比喩、対応表、比喩の限界、技術本体へ戻す一文を書く。

比喩は、一語対一語の置き換えより、複数の登場役、道具、時間の流れを持つシーンとして考える。たとえば「RAG = カンペ」だけで止めず、「台本や資料を読んでこなかった生放送の出演者に、裏方スタッフが本番中に必要なカンペを出す」のように、構造が対応する場面を作る。

この工程は、本文にそのまま出す制作メモではなく、読者が追える説明を作るための設計図である。`draft-0x.md` では、採用したシーン比喩を短く使い、比喩のあとに技術本体へ戻る文を置く。

## レビューの流れ

1. `draft-0x.md` をレビューする。
2. 良い点、改善点、`abstract-0x.md` に戻す内容、`draft-0x.md` での反映方針を `review-0x.md` に書く。
3. `review-0x.md` をもとに、まず `abstract-0x.md` を更新する。
4. 更新後の `abstract-0x.md` と `review-0x.md` をもとに、`draft-0x.md` を更新する。

`review-0x.md` は判断ログとして使う。本文そのものの再執筆は `draft-0x.md` に反映する。

レビュー反映に入る前に、各ページごとに必要な編集レベルを判定する。編集レベルは、現在の本文をどれだけ残すかではなく、解説書としての品質を上げるために必要な作業量で選ぶ。

編集レベルの選択肢:

- オーバーホール: ページの核、説明順、比喩、図の設計を `abstract-0x.md` から組み直す。
- 半分程度の再執筆: 良い節を残しつつ、ページの主説明や後半の流れを大きく書き直す。
- 1-2割の再執筆: 主要な流れは残し、弱い節、接続、回収、具体例を差し替える。
- 軽微な本文修正: 文言、順序、短い補足、用語説明を直す。
- 図の追加: 本文の説明密度を上げるため、図を1点以上追加する。
- 図の入れ替え: 本文は維持し、誤解を招く図、弱い図、説明とずれた図を差し替える。
- 記録のみ: 現時点では本文に反映せず、次の改稿候補として残す。

`review-0x.md` には、選んだ編集レベル、選定理由、`abstract-0x.md` に戻す内容、`draft-0x.md` で反映する内容、図で対応する内容を分けて書く。

レビューでは、`project.md`、`series_plan.md`、`handoff.md` と、この文書に登録されたレビュー観点文書を選んで使う。観点ごとの細かな基準は各レビュー観点文書に置き、ここでは適用手順を扱う。

この `policies/` フォルダには、レビュー観点として `reader-conviction-perspective.md` と `intellectual-delight-perspective.md` がある。

- `reader-conviction-perspective.md`: 読者の納得感を作る観点。読者の実感、読者の足場、読者の頭の中にできる絵を見る。
- `intellectual-delight-perspective.md`: 学び直しの知的快感を作る観点。予想の更新、小さな疑問と回収、読後の見え直しを見る。
- `final-quality-checklist.md`: ドラフト完成後に商品価値を上げる観点。高度な中身、説明できる教養、具体例と比喩の効き方、ページごとの情報密度を見る。

各観点文書は、候補抽出と判定の両方で使う。候補抽出では、その観点で気になる箇所を広く拾う。判定では、各候補を「改善採用」「増強採用」「記録のみ」に分ける。

ポジティブな候補は、本文にすでに効いている箇所として拾う。判定では、元の良さの大きさより、編集による伸び幅を見る。本文の一文、図の作り方、前後の回収、短文リズムなど、知的快感を高める具体案が書ける候補を「増強採用」にする。具体案が薄い良い点は、「記録のみ」にする。

判定基準を変えたときは、旧判定と新判定を同じ重みで並べず、`review-0x.md` を最新版として整理する。判断履歴を残す場合も、最新判定がどれかを明確にする。

レビュー観点文書を追加したときは、この節にリンクと用途を追記する。

各レビュー観点は、次の順で `review-0x.md` に適用する。

1. 候補抽出: 観点文書に沿って、本文から気になる箇所を広めに拾う。候補箇所、候補表現、候補にした理由を書く。
2. 判定: 各候補について、観点文書の基準で「改善採用」「増強採用」「記録のみ」を決める。
3. 改善採用: 読者が追える表現や説明順への直し方を書く。
4. 増強採用: すでに効いている箇所について、知的快感を高める具体的な編集案を書く。
5. 記録のみ: 候補として拾い、本文への反映を保留する箇所について、判断理由を書く。
6. 反映方針: `abstract-0x.md` に戻す内容と、`draft-0x.md` で直す内容、本文で強める内容を分けて書く。

複数のレビュー観点を使う場合は、観点ごとに節を分ける。候補抽出と判定を分けることで、LLMが一度の出力で指摘を決め打ちする揺れを抑える。

## 前ページの参照

`abstract-0x.md`、`draft-0x.md`、その他のページ番号つきファイルを書くときは、それまでの流れを踏まえてよい。

たとえば `abstract-03.md` を書くときは、`abstract-01.md`、`draft-01.md`、`abstract-02.md`、`draft-02.md` を参照してよい。前のページで読者が何を知ったかを確認しながら、説明のつながりを作る。

## draft-0x.md を書くときの方針

- `draft-0x.md` は、対応する `abstract-0x.md` を入力として書く。
- `abstract-0x.md` に書いた「説明すること」は、本文の流れに自然に入れる。
- `abstract-0x.md` に書いた「図示すること」は、本文中に `[ここに「...」図を入れる]` の形で置く。
- `abstract-0x.md` の本文行数目安に沿って、1ページあたり本文50行程度、約1000字程度を目安にする。本文行は、ページ見出し、空行、図指定、コードブロックや `text` ブロックなどの例示枠を除いて数える。
- ページタイトルは、本文の面白さを先取りするより、そのページで説明する本質を示す。必要なら専門用語も使い、本文で身近な例と図解に戻す。専門語を出すときは、そのページの説明トーンに合い、読者の足場から自然に進める理由があるかを見る。
- 本文の節見出しは、`abstract-0x.md` で計画した説明のまとまりに合わせて置く。見出しは読者が流れをつかむために使い、本文そのものは見出しに頼りすぎず自然につなげる。
- 新しい用語は、読者が見ているもの、具体例、操作、例え話で中身を先に説明してから名前に結びつける。用語名を先に置く場合も、直後に読者の足場から意味をつかめる説明を置く。
- UTF-8、RGB、JPEG、圧縮のように、読者が普段目にする語の仕組みを説明できる場合は積極的に使う。名前に続けて、「何の約束か」「何を変換しているのか」まで本文で一口ぶん説明する。
- 比喩や身近な例は、高度な中身へ届く橋として使う。すでに易しい内容を易しく言うだけで満足せず、読者が聞いたことのある言葉の背後にある設計、制約、トレードオフへ踏み込む。
- 「中身や仕組みがわかった楽しさ」は、ページ末尾と説明の途中の両方ににじませる。
- 短文のリズムを残す箇所と、普通に一段落で読ませる箇所を分ける。要点の転換、驚き、回収は短文で一拍置き、同じ説明の中で自然につながる文は段落として読ませる。
- 各ページには、そのページで説明する核をひとつ置く。後続ページで扱う説明は、後続ページへ送る。

## 本文上の注意

- 「図解案」「つまずきやすいポイント」「ChatGPT深堀」のような制作メモは、10ページ本文の後に置く付録や制作メモとして扱う。
- 図は1ページ4-5点を標準にする。
- 図を減らす場合も、本文ページは、図や例示枠を除いた本文だけで平均50行程度、約1000字程度を目安にする。
- 1文は20字程度を目安にする。短く言い切る文は10-20字、説明のために少し伸ばす文でも30字台前半までに抑える。
- 語り口は、`project.md` の執筆方針に合わせて、ですます調で書く。
- 専門語は必要なときだけ使い、初出ではやさしく言い換える。
- 本文では「一列に並べる」「順番に並んでいる」のように説明する。

## 最終品質チェック

ドラフト完成後、商品価値を上げる推敲に入る前に、`final-quality-checklist.md` を使う。

チェックリストでは、各ページの現状を採点するより、伸ばす余地の大きい箇所を見つける。チェック結果は、次の推敲で `abstract-0x.md` に戻す内容、本文で加筆する内容、図で補う内容に分ける。
