# 10ページ大学 情報学部 第4巻 ページ構成案

## 巻タイトル

**インターネットとクラウドが１秒間でやっていること**

## サブタイトル案

**スマホで結果が返るまでの道のりを、10ページでつかむ**

## この巻の位置づけ

第1巻では、半導体のスイッチが0と1を作り、その0と1で計算・記憶・命令実行ができることを扱った。

第2巻では、文字・画像・音声・動画が、0と1のデータとして表せることを扱った。

第3巻では、そのデータをアプリやOSがどう扱い、画面・ファイル・デバイスとつなぐかを扱った。

第4巻では、その続きとして、アプリが作ったデータが端末の外へ出て、インターネットを通り、クラウド上のサーバーへ届き、結果として画面に戻るまでを扱う。

この巻の中心テーマは、次の一文である。

> Webサービスでボタンを押して結果が返る体験は、端末、近くのネットワーク、DNS、IPアドレス、パケット、ルーティング、TCP/UDP、HTTP/HTTPS、Web API、クラウド側の役割分担が順番につながって成立している。

第4巻の本文は、この巻だけで読後感が閉じる構成にする。AIアプリやWebサービスが「遠くの計算機へお願いし、結果を受け取っている」という土台を作り、次巻への接続は10ページ目の最後に限る。

## 構成方針

この巻は、操作が結果に変わる順番で構成し、用語はその流れの中で置く。

はじめにと1ページ目を導入部として使い、読者が普段見ているWebサービスの体験から入る。たとえば、スマホで地図アプリを開き、近くの店を検索し、写真や口コミや経路が表示される場面を足場にする。ChatGPTのようなAIアプリに質問する場面も、同じ流れの先に置く。

本文10ページは、ひとつの操作が結果に変わるまでの順番で進める。各ページは、前ページで生まれた疑問や必要性から始める。

- 1ページ目: ボタン操作は、サーバーへのお願いに変わる。
- 2ページ目: そのお願いは、まず近くのネットワークから外へ出る必要がある。
- 3ページ目: 人間が覚える名前を、機械が使う住所へ変える。
- 4ページ目: 大きなデータは、小さく分けて運べる形にする必要がある。
- 5ページ目: 小さなデータは、ネットワーク同士の接続点を渡って進む必要がある。
- 6ページ目: 相手のコンピュータへ届いたあと、どの窓口へ渡すかを決める必要がある。
- 7ページ目: サーバーと何をどう頼み、何を返すかという会話の形が必要である。
- 8ページ目: クラウド側では、認証、保存、検索、計算、配信が分担して返事を作る。
- 9ページ目: 多くの人へ速く返すために、近くに置く、分けて受ける設計が必要になる。
- 10ページ目: 止まりにくく、増やしやすい設計まで含めて全体を回収し、最後にだけAIへつなぐ。

各ページの冒頭では、前ページの操作が次へ進むためにどの仕組みを必要とするかを短く置く。未知寄りの仕組みでは、「それがあると何ができるようになるか」を先に置く。

### 深掘り方針

この巻では、読者の見え方が変わる説明を先に作り、専門用語のラベルはそのあとで置く。DNS、パケット、ルーティング、TCP、HTTP、クラウド、CDN、冗長化は、一般的な一文説明で終わると商品価値が小さい。

各ページでは、次のうち少なくとも一つへ踏み込む。

- 設計上の制約: なぜその形が必要になったのか。
- 読者の日常から立つwhy: 普段見る画面、設定、契約、待ち時間、トラブル、同じ操作なのに結果が違う場面から必要性を立てる。
- 内部の約束: どの情報をどの層や番号で扱うのか。
- トレードオフ: 速さ、確実さ、鮮度、コストのどれを優先しているのか。
- 失敗時の見え方: どこで失敗すると画面や体験にどう出るのか。
- 運用上の判断: 期限、切り替え、監視、負荷、状態をどう扱うのか。

ページ別の深掘り芯は次の通り。

- 1ページ目: リクエストは、操作の種類、宛先、本文、本人確認の印などを分けて持つ。
- 2ページ目: Wi-Fi、家庭内ネットワーク、ルーター、外向き回線の境界を見る。
- 3ページ目: DNSは、同じURLやアプリ名で開ける期待を支えつつ管理者ごとに住所変更を扱うため、階層とキャッシュ、TTLで成り立つ。
- 4ページ目: パケットはMTUの制約に合わせ、IPやTCP/UDPのヘッダーが層として重なる。
- 5ページ目: インターネットは、家、外出先、会社で入口が変わる体験の延長にある。管理者の違うAS単位のネットワークが届く範囲を知らせ合い、ルーターは巨大な表を扱うため宛先範囲で次を選ぶ。
- 6ページ目: TCPは再送、順序制御、輻輳制御を持ち、UDPは確認を上位へ委ねやすい。
- 7ページ目: HTTPは、検索、投稿、写真追加、ログインのように同じ画面内のボタンでも頼みごとが違う体験から、メソッド、パス、ヘッダー、本文、状態コードへ分かれる必要を見せる。HTTPSは途中のネットワークを通る通信に相手確認と保護を重ねる。
- 8ページ目: クラウドは、店カードにまとまって見える情報や写真だけ遅れて出る体験を足場に、情報量、計算量、同時利用を分けて扱う受付、認証、索引検索、オブジェクト保存、キュー、ワーカーの分担として見る。
- 9ページ目: キャッシュとCDNは、地図画像や店の写真がすぐ出る体験から、遠さと集中を減らしつつ鮮度を扱う仕組みとして見せる。
- 10ページ目: 冗長化は、セール開始や新作公開でサービスが重くなる体験を足場に、予備だけでなく、監視、切り替え、状態管理を含むものとして扱う。

## この冊子でわかること

- スマホで検索、投稿、ログイン、AIへの質問をするとき、端末が外のサーバーへリクエストを送っていること
- Wi-Fi、ルーター、プロバイダ、モバイル回線が、端末から外へ出る最初の道になること
- URLやアプリの宛先名が、DNSによってIPアドレスへ変わること
- データがパケットという小さな単位に分かれ、宛先や送り主の手がかりを持って運ばれること
- インターネットが、ネットワーク同士の接続と中継で遠くのサーバーまで道を作ること
- TCP、UDP、ポートが、届き方と届け先の窓口を支えること
- HTTP、HTTPS、Web APIが、アプリやブラウザとサーバーの会話を作ること
- クラウド側で、認証、データベース、ストレージ、計算、配信が役割分担していること
- キャッシュ、CDN、負荷分散、冗長化が、速く止まりにくいサービスを作ること
- AIアプリも、こうした通信とクラウドの上で動いている場合が多いこと

## 読後の到達点

読者は、以下をざっくり説明できるようになる。

- Webサービスでボタンを押すと、端末からサーバーへリクエストが送られ、レスポンスが戻ること
- Wi-Fiやモバイル回線は、インターネットへ出る最初の入口であること
- ブラウザならURL、アプリならAPIの宛先名を、DNSでIPアドレスへ変えてから通信すること
- 大きなデータがパケットに分かれて、ルーターを通って運ばれること
- インターネットは、ネットワーク同士の接続でできていること
- TCPとUDPの違いを、確認を組み込む通信と、確認を減らしてリアルタイム性を優先しやすい通信の違いとして説明できること
- HTTPSが、通信内容を暗号化し、相手の確認を助けること
- Web APIが、データをやりとりするためのサーバーとの窓口であること
- クラウドが、データセンター上のコンピュータに計算、保存、検索、配信を任せる仕組みであること
- 速いWebサービスの裏側に、キャッシュ、CDN、負荷分散、冗長化があること

## はじめにの設計

### 役割

はじめには、本文10ページの前に読者の体験をそろえる導入として使う。商品のパッケージにあたるため、短く、簡潔に、この巻を読む面白さを提示する。

中心シーンは、スマホでWebサービスを使う場面にする。たとえば、検索ボタンを押して結果が返る、ChatGPTに質問する、写真をクラウドに保存する、という日常の操作を短く並べる。

画面上では「ボタンを押したら結果が出る」だけに見える。はじめにでは、その短い待ち時間の裏側で、端末、通信、サーバー、クラウドが順番に働くことを示す。

サーバーははじめにで早く出る言葉なので、初出直後に「お願いを受け、返事を作る相手」として仮置きする。はじめにではサーバー説明を短く置く。1ページ目でも役割の仮置きに留め、8ページ目でクラウド側の役割分担として実体を回収する。

はじめにでは、本文10ページの全体マップを先に図示する。読者が各ページを読む前に、「いま読んでいるページは、Webサービス利用のどの地点を説明しているのか」を戻って確認できるようにする。

本文は短く絞る。導入の役割は、読者に「この1秒ほどの裏側を見てみたい」と思わせることに絞る。

### 本文で置く問い

- この1秒ほどの待ち時間に、何がどこまで行って戻っているのか。
- Wi-Fiにつながっているのにページが開かないとき、どこで止まっているのか。
- 聞いたことはあるが説明しにくい言葉が、一本の道としてどうつながるのか。

### 全体マップの設計

はじめにの主図として、次の流れを横長の道筋で見せる。

`端末の操作 → リクエスト → 近くの入口 → DNS → パケット → ルーター群 → ポート/TCP/UDP → HTTP/HTTPS/Web API → クラウドの役割分担 → キャッシュ/CDN/負荷分散 → 冗長化/仮想化 → レスポンス → 画面更新`

図の各地点に、対応するページ番号を短く付ける。

- 1ページ目: 操作がリクエストになる
- 2ページ目: 近くの入口から外へ出る
- 3ページ目: DNSが名前を住所へ変える
- 4ページ目: データをパケットに分ける
- 5ページ目: ルーターが次へ渡す
- 6ページ目: ポートとTCP/UDPで窓口と届き方を決める
- 7ページ目: HTTP/HTTPS/Web APIで会話する
- 8ページ目: クラウド側で認証、保存、検索、計算、配信を分担する
- 9ページ目: キャッシュ、CDN、負荷分散で速く返す
- 10ページ目: 冗長化、仮想化、スケールで止まりにくく増やしやすくする

この図は、各ページの詳細の先取りを抑える。ページ番号、短いラベル、流れの向きだけを見せる。本文では、前ページで生まれた次の必要性から入り、必要に応じてこの全体マップへ戻って位置を確認できるようにする。

## 1ページ目：ボタンを押すと、サーバーへのお願いが生まれる

### ストーリー上の役割

はじめにと1ページ目を導入部として扱う。画面上の体験を、通信の最小単位であるリクエストとレスポンスへ変換する。

### 必要性の入口

画面上では「検索」「送信」「ログイン」「質問」を押しただけに見える。しかし結果を返すには、端末の外にある相手へ、何がほしいのかを伝える必要がある。

### 伝える要点

- 第3巻で見たアプリやOSの続きとして、通信を見る
- アプリの操作は、外のサーバーへ送るリクエストに変わる
- サーバーへ送るお願いを、リクエストと呼ぶ
- サーバーから返ってくる返事を、レスポンスと呼ぶ
- ボタン操作は、URL、検索語、本文、認証情報、設定値などの小さなデータに変わって行き来する
- 端末側はクライアント、向こう側はサーバーと呼べる
- サーバーは、この時点では「返事を作る相手」として仮置きし、実体は8ページ目で分解する

### 説明の方向

地図アプリで「近くのカフェ」と検索する場面から入る。画面上では検索語を入れてボタンを押しただけに見える。そこで、その裏側では、アプリが端末の基本機能をまとめるOSの通信機能を使い、外のサーバーへリクエストを送り、レスポンスを受け取って画面を書き換えていると説明する。

DNSやTCPの詳細は後続ページへ送る。まず、画面の操作が「外の相手との会話」へ変わるという見方を作る。

### ページ末の次へのつなぎ

画面の操作が外へ送るデータになったことを回収し、次はそのデータを近くの入口へ渡す最初の受け渡しへ進む。

### 使う比喩

- クライアント：注文する客
- サーバー：注文を受けて返事をする店
- リクエスト：注文票
- レスポンス：返事と中身

比喩のあとで、注文票はあくまで説明の足場であり、実際には決められた形式のデータが送られると戻す。

## 2ページ目：お願いは、まず近くの入口から外へ出る

### ストーリー上の役割

1ページ目で生まれたリクエストが、端末の外へ出る最初の一歩を見る。

### 必要性の入口

リクエストは、まず端末の近くにあるネットワークへ渡される。「Wi-Fiマークは出ているのにページが開かない」という経験は、この入口とその先を分けて見るための足場になる。

### 伝える要点

- スマホやパソコンは、まず近くのネットワークにつながる
- 家や店ではWi-Fiが最初の入口になることが多い
- Wi-Fiは無線で近くの機器につながる仕組みであり、端末から近くの入口までを示す
- ルーターは、家や店の中の機器を外のネットワークへつなぐ出入口である
- 家の中の複数端末は家庭内ネットワークに入り、ルーターが内側と外側の境界として中継する
- 申し込みや請求では一つに見える通信プランも、Wi-Fi、回線、プロバイダ、通信会社の役割へ分けて考えられる
- 光回線は、家や店までの回線と、その先をインターネットへ接続するプロバイダの役割が重なることで使える
- モバイル回線は、基地局と通信会社のネットワークが入口になる
- 端末から外へ出るまでにも、すでにいくつかの中継がある

### 説明の方向

「Wi-Fiはつながっているのにインターネットが見られない」という経験を入口にする。Wi-Fiへ接続している状態と、インターネットの先へ出られる状態を分けて見せる。

Wi-Fiの説明は、端末から近くの機器までの無線区間に絞る。Wi-Fiマークや無線接続そのものは読者にとって既知寄りなので、紙幅は「Wi-Fiの先」にある回線、プロバイダ、通信会社の分担へ使う。

家庭の例では、スマホ → Wi-Fiルーター → 回線を外へ渡す機器 → プロバイダへ進む。機器名は環境によってONUやモデムなどと呼ばれることがある。家の中では複数の端末が家庭内ネットワークに入り、ルーターが内側と外側の境界として外向き通信を中継する。外出先では、スマホ → 基地局 → 通信会社のネットワークへ進む。

自宅やカフェで見かける機器を足場にしつつ、商品や契約の見え方と仕組み上の役割を分ける。セットプランや同じブランドで一体に見える場合でも、通信や回線の契約、プロバイダ契約、Wi-Fi機器、通信会社のネットワークは役割として分けて扱う。料金プランや契約法務の細部は背景に留め、読者が「Wi-Fiが悪い」と一語で見ていた不調を、入口、回線、プロバイダ、通信会社へ切り分けられるところまでを狙う。

### ページ末の次へのつなぎ

外へ出る入口が見えたら、次は名前と住所の変換へ進む。人間が入力する名前を、機械が使う住所へ変える。

### 使う比喩

- Wi-Fi：部屋の中の無線の通路

## 9ページ目：速く返すために、近くに置き、分けて受ける

### ストーリー上の役割

クラウド側で返事を作れるようになったあと、多くの人に速く返す設計を見る。

### 必要性の入口

アプリを開いたとき、地図の細かい画像や店の写真がすぐ出ると、地図を拡大したり店を選んだりする動きが続く。人気のあるWebサービスでは、同じ画像、同じ動画、同じページ、同じ地図データを多くの人が見る。毎回遠くの中心サーバーまで取りに行くと遅くなり、同じ入口に人が集中すると詰まりやすい。

### 伝える要点

- キャッシュは、よく使うデータを近くに一時保存する仕組みである
- CDNは、世界各地にデータのコピーを置き、近い場所から届ける仕組みである
- 負荷分散は、たくさんのアクセスを複数のサーバーへ分ける仕組みである
- 地図画像、動画、商品画像、アプリの静的な部品は、近い場所から返せると速くなる
- すべてのリクエストを同じサーバーで処理するより、役割や混み具合に応じて分けるほうが止まりにくい
- キャッシュやCDNは、コピーや一時保存をうまく使う設計である

### 説明の方向

地図アプリの検索結果で、地図タイルや店の写真が素早く出る場面から入る。毎回遠くのデータセンターからすべてを取り直すより、近い場所に置いたコピーから返すほうが速い。

負荷分散は、同じサービスへ多くのリクエストが来たとき、複数のサーバーへ振り分ける仕組みとして説明する。5ページ目で見た途中の道選びとは範囲を分け、サーバー側の入口で分ける話として扱う。

### ページ末の次へのつなぎ

速く返せるようになっても、機械は壊れ、アクセスは増減する。最後に、止まりにくく増やしやすくする設計と、AIアプリへのつながりを回収する。

### 使う比喩

- キャッシュ：よく使う資料を机の上に置いておく
- CDN：遠い本店だけでなく、近くの支店から届ける
- 負荷分散：混んだ受付を複数の窓口に分ける

比喩のあとで、実際にはデータのコピー、一時保存、振り分けのルールでレスポンスを速くしていると戻す。

### 誤解防止の扱い

- CDNやキャッシュは、元データと配信用コピーの関係として扱う
- キャッシュは、期限、再確認、鮮度管理を含む仕組みとして扱う
- 負荷分散はサーバー側の受け口の分け方、ルーティングは途中経路の選び方として分ける

## 10ページ目：止まりにくく、増やしやすい設計からAIへつながる

### ストーリー上の役割

一度のWebサービス利用を最初から最後まで回収し、最後にだけクラウド上のAIへ接続する。

### 必要性の入口

セール開始、新作公開、災害時の情報確認では、同じサービスへ人が一気に集まる。ふだん使えるアプリが、その時間だけ重くなることもある。Webサービスでは、速さに加えて、止まりにくさと増やしやすさも必要になる。機械が壊れる日も、アクセスが急に増える日も、更新する日もある。多くの人が使うサービスには、止まりにくく、増やしやすく、直しやすい設計が必要になる。

### 伝える要点

- 冗長化は、壊れたときのために予備を持つ設計である
- 仮想化により、1台の実物のサーバー、つまり物理サーバーを複数の仮想的なコンピュータとして使えることがある
- クラウドでは、必要に応じてサーバーや保存領域を増やしたり減らしたりできる
- 速さ、止まりにくさ、コストにはトレードオフがある
- Webサービスは、通信、計算、保存、配信、障害対策の組み合わせで動いている
- AIアプリも、こうした通信とクラウドの上で動いている場合が多い

### 説明の方向

第4巻のまとめとして、通信の流れをもう一度たどる。

スマホのアプリがリクエストを作る。近くのネットワークから外へ出る。DNSで名前を調べる。データはパケットになり、ルーターを通る。ポートとTCP/UDPの約束で相手の窓口へ届く。HTTP/HTTPSやWeb APIの形でサーバーと会話する。クラウド側では、認証、データベース、ストレージ、計算、配信が役割分担する。キャッシュ、CDN、負荷分散、冗長化で速く止まりにくくする。

最後に、AIアプリへの導線を作る。ChatGPTのようなAIアプリに質問すると、端末からクラウド上のサーバーへ通信して処理される場合がある。第5巻では、その先でAIが文章・画像・音声をどう扱うかへ進む。

### 使う比喩

- 冗長化：予備の橋や予備の発電機を用意する
- 仮想化：大きな建物を部屋に区切って貸すこと
- スケール：混み具合に合わせて窓口や作業場を増やすこと

比喩のあとで、実際には実物のサーバー、仮想的なコンピュータ、保存領域、監視、切り替えの設計で支えていると戻す。

- [ここに「仮想化で1台を複数の仮想的なコンピュータとして使う」図を入れる]
- [ここに「端末 → 入口 → DNS → パケット → ルーター → HTTPS/Web API → クラウド → レスポンス」の全体図を入れる]
- [ここに「第5巻はクラウド上のAIへ進む」図を入れる]

### 誤解防止の扱い

- クラウドは、監視、切り替え、状態管理で止まりにくさを作る仕組みとして書く
- 仮想化やスケールは、費用と管理を伴う増減の仕組みとして書く
- AIの仕組みは第5巻への橋に留める

## 理解確認クイズ候補

1. Webサービスで検索ボタンを押すと、リクエストとレスポンスはどのように関係しますか。
2. Wi-Fiにつながっているのにページが開かないとき、Wi-Fiとインターネットの違いから何を考えられますか。
3. DNSは、URLやアプリの宛先名に対して何を調べますか。
4. パケットとは何ですか。大きなデータを分けて送る理由を説明してください。
5. ルーターは、パケットを受け取ったあと何を見て次の行き先を決めますか。
6. ポート番号は、IPアドレスに加えて何を補いますか。
7. TCPとUDPの違いを、目的の違いとして説明してください。
8. HTTPSの鍵マークは、何を示し、どの確認が別に必要ですか。
9. Web APIは、ブラウザで見るWebページと何が違いますか。
10. クラウド側で、認証、データベース、ストレージ、計算、配信が分かれている理由を説明してください。

## ChatGPT深掘りプロンプト候補

1. スマホでWebサービスの検索ボタンを押して結果が返るまでを、リクエスト、DNS、パケット、HTTP、クラウドを使って順番に説明してください。
2. Wi-Fi、ルーター、プロバイダ、インターネットの違いを、端末から外へ出る最初の道として説明してください。
3. DNSが必要になる理由を、URLやアプリの宛先名とIPアドレスの違いから説明してください。
4. パケット通信を、1枚の写真を小さな封筒に分けて送る比喩で説明してください。
5. インターネットがネットワーク同士のつながりであることと、ルーティングの関係を説明してください。
6. TCP、UDP、ポート番号の違いを、届き方と窓口の違いとして説明してください。
7. サーバーへ送るお願いと返事の中身を、注文票と返事の比喩で説明してください。
8. HTTPSが暗号化と相手確認をする仕組みを、専門用語を減らして説明してください。
9. クラウド側で、アプリサーバー、認証、データベース、ストレージがどう分担するかを説明してください。
10. CDN、キャッシュ、負荷分散、冗長化がWebサービスを速く止まりにくくする理由を説明してください。
