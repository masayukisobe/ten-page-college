# 7ページ目：サーバーと何を話すか：HTTP・HTTPS・Web API

## 届いたデータを読める形にする

6ページ目で、相手の窓口と届き方を見ました。

次に必要なのは、サーバーと何をどう話すかです。

地図アプリでは、検索ボタン、口コミ投稿、写真追加、ログインが同じ画面の中に並びます。

利用者から見ると、どれもボタン操作です。

サーバー側から見ると、動かす仕事は変わります。

「このページをください」。

「この検索結果をください」。

「この投稿を保存してください」。

そうしたお願いを、サーバーが読める形にする約束が必要です。

1ページ目のリクエストとレスポンスは、ここで具体的な形を持ちます。

## HTTPリクエストの中身

HTTPは、ブラウザやアプリがサーバーと会話するための約束です。

サーバーは、届いたデータを見て、どの処理を動かすかを選びます。

同じ店情報の宛先でも、一覧を見たいのか、口コミを投稿したいのか、写真を追加したいのかで作業が変わります。

その違いを、機械が同じ読み方で判定できる形にします。

HTTPのリクエストには、いくつかの部品があります。

どんな操作かを示すメソッド。

どの場所やデータがほしいかを示すパス。

追加情報を入れるヘッダー。

検索条件や投稿内容のような本文。

たとえば同じ `/shops` という宛先でも、一覧を取る操作と、新しい投稿を保存する操作は分かれます。

ヘッダーには、受け取りたいデータ形式、ログインに関係する印、アプリの情報などが入ることがあります。

HTTPは「会話」というより、サーバーが機械的に読める注文票の部品を決める約束です。

## レスポンスは状態と中身を返す

サーバーは、リクエストを読んだあと、レスポンスを返します。

レスポンスには、結果の状態が入ります。

うまくいった。

対象なし。

ログインが必要。

権限不足。

サーバー側で失敗した。

こうした状態を番号で示すものが、状態コードです。

エラー画面にも、失敗の種類を伝える返事が含まれます。

中身としては、Webページを組み立てるHTML、画像、アプリが読みやすいJSONなどが返ります。

ブラウザはHTMLを読んでページを組み立てます。

スマホアプリはJSONのようなデータを読んで、自分の画面を組み立てることがあります。

## HTTPSは相手確認と暗号化を重ねる

Webページのアドレスで、鍵のアイコンを見ることがあります。

これはHTTPSの手がかりです。

ログイン、決済、投稿の内容は、家のルーター、通信会社、途中のネットワークを通って相手へ進みます。

途中の機械には、次へ運ぶ仕事があります。

一方で、ログインや投稿の中身は、相手のサーバーへ届くまで守りたい情報です。

HTTPSは、HTTPの通信に保護された通り道を重ねます。

その準備には、TLSという仕組みが使われます。

サーバーは、自分がそのドメインの相手だと示す証明書を出します。

ブラウザやアプリは、その証明書を手がかりに相手を確認し、暗号化に使う鍵の準備をします。

暗号化されると、途中の機械が通信内容の意味を読み取りにくくなります。

鍵マークが主に示すのは、通信路の保護と相手確認です。

## Web APIは構造化データを返す窓口

サーバーとの会話は、ブラウザで見るWebページにも、スマホアプリの裏側にもあります。

スマホアプリがサーバーとデータをやりとりする窓口を、Web APIと呼びます。

Web APIでは、画面そのものより、アプリが読める構造化データが返ることが多いです。

天気アプリは天気データを受け取る。

地図アプリは検索結果や地図データを受け取る。

AIアプリは質問を送り、返答を受け取る。

アプリは返ってきたデータを、自分の画面へ並べ直します。

返ってくる材料は、画面を作るためのデータです。
