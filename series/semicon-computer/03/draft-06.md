# 6ページ目：画面表示とGUI：ボタンは絵と状態と処理でできている

## 画面のボタンは、領域と約束で動く

スマホの画面には、ボタンが出ます。

送信ボタン、戻るボタン、再生ボタン。

押すと反応します。

画面のボタンには、押せそうに描かれた見た目があります。

アプリは、その絵が出ている場所を、押せる領域として持っています。

その領域にタップが届いたとき、送信する、戻る、再生する処理へ進む約束も持っています。

四角、文字、色、影は、人に「ここを押せる」と知らせる印です。

押せる領域は、コンピュータが「ここが押された」と判定するための範囲です。

処理への約束は、押されたあとに何をするかを決めます。

画面の部品は、見た目、領域、処理への入口が結びついています。

![ボタンの見た目、押せる領域、処理への約束が結びつく](images/fig-06-01-button-region-promise.jpg)

## GUIは見て操作する仕組み

GUIという言葉があります。

Graphical User Interfaceの略です。

グラフィカルな画面を使って、人がコンピュータを操作する仕組みです。

ボタン、メニュー、ウィンドウ、入力欄。

こうした部品を見て、押したり選んだりします。

文字だけで命令を打つ操作とは違います。

読者がふだん使うスマホやPCの多くは、このGUIを使っています。

GUIは、見た目がきれいな画面というだけではありません。

人が見て選べる部品で、操作を組み立てる仕組みです。

![GUIはボタン、メニュー、入力欄で操作する画面](images/fig-06-02-gui-controls.jpg)

## アプリは部品の配置を決める

GUIの中身を組み立てるのは、主にアプリ側の仕事です。

アプリは、画面に何を出すかを決めます。

タイトルをどこに置くか。

写真をどの大きさで出すか。

ボタンをどこに置くか。

文字入力欄をどう見せるか。

その指示が、OSや描画の仕組みを通って画面に出ます。

OSは、アプリに画面を出す場所や描画の窓口を用意します。

この窓口があると、アプリは画面の出し方を共通の形で頼めます。

画面同士の重なり方もOS側で扱いやすくなります。

ディスプレイは、最後にピクセルとして表示します。

第2巻では、画面の最後はピクセルとして表示されると見ました。

ここでは、その手前を見ています。

アプリは、ピクセルを一つずつ手で塗るより、部品や状態を使って画面を組み立てます。

![アプリの部品配置からOSや描画処理を通ってピクセル表示になる](images/fig-06-03-layout-to-pixels.jpg)

## 画面には状態がある

画面は、一度描いたら終わりではありません。

状態が変わるたびに、描き直されます。

入力欄に文字が入った。

チェックが入った。

スクロール位置が変わった。

写真の明るさが変わった。

こうした現在の値を、状態と呼ぶことがあります。

画面は、状態をもとに描かれます。

状態が変わると、見た目も変わります。

ホワイトボードに、今の状況を書き直していくようなものです。

![状態が変わると画面が描き直される](images/fig-06-04-state-redraw.jpg)

## ボタンは見た目、領域、役割、処理でできている

ボタンを一つ見ても、中にはいくつかの要素があります。

まず、見た目です。

押せそうな形、色、文字があります。

次に、押せる領域です。

画面のどの範囲を押したら、そのボタンとして扱うかが決まっています。

次に、役割です。

送信する、保存する、戻る、といった意味があります。

最後に、押されたときの処理です。

これらが結びついて、画面のボタンになります。

絵に押せる領域と処理が結びついて、はじめて操作できるボタンになります。

処理も、画面に見える形や押せる範囲と結びつくことで、人が使える操作になります。

画面が固まる、崩れる、読み込み中になる。

そうした現象も、状態や描画や処理のつながりで見ると、少し整理できます。

そして、ボタンが本当に動くには、もう一つ必要なものがあります。

押されたことが、アプリへ届く必要があります。
