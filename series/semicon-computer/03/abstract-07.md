## 構成、節見出し、行数目安

### 1. タップは位置と時刻を持つ知らせになる: 10-12本文行

- タップから入り、タッチパネルが触れられた位置の変化を検出し、OSが時刻などの情報を添えてアプリへ届ける知らせに整えると説明する。
- OSは入力装置から来た変化を受け取り、どのアプリ、どのプロセスへ届けるかを決める。
- イベントを「何が、いつ、どこで起きたか」の情報として定義する。

### 2. イベントは出来事の注文票: 10-12本文行

- イベントを受付に届く注文票にたとえる。
- 種類、時刻、場所、対象を持つことを説明する。

### 3. 座標から押された部品を判断する: 10-12本文行

- タップ座標と画面部品の位置を照らし合わせ、どの部品に起きた出来事か判断する。
- OSはまず操作対象のアプリを見つけ、アプリは自分のGUIの中で部品を判断する。
- 同じタップでも場所で意味が変わることを示す。

### 4. アプリは待ち、処理し、更新する: 12-14本文行

- 入力を待つ、イベントを受け取る、処理する、状態を変える、画面を更新する流れ。
- イベントループを入口だけ扱う。
- OSは入力を届け、アプリはイベントの意味を決めて処理するという分担を明確にする。

### 5. 反応が遅いとき: 10-12本文行

- 反応遅延の原因は複数あると置く。
- イベント処理が滞ると、押したのに届いていないように見えると回収する。

## 説明すること

- キーボード、マウス、タッチパネルからの入力は、イベントとしてアプリに届く。
- イベントは、種類、時刻、場所、対象などを持つ出来事の知らせである。
- アプリはイベントを待ち、受け取り、処理し、画面を更新する。
- OSは、入力装置から来た情報をイベントとして整え、対象アプリのプロセスへ届ける。
- 反応が遅い現象は、この流れのどこかの詰まりとして考えられる。

## 構造分析とシーン比喩

- 概念・用語・仕組み: 入力装置、イベント、座標、対象部品、イベントループ、画面更新。
- 理解上昇の入口（未知寄り）: OSが入力をイベントとして整えることで、タッチ、クリック、キー入力をアプリが同じ「出来事の知らせ」として受け取れると説明する。
- 役割: 入力装置は変化を検出し、OSは整理し、アプリはイベントを処理する。
- 情報の流れ: 指やキー操作 -> 入力装置の検出 -> OSの整理 -> イベント -> アプリ処理 -> 状態変化 -> 画面更新。
- 時間順の変化: 待つ -> 届く -> 判断する -> 処理する -> 更新する -> また待つ。
- 失敗パターン: 「指の力が直接入らない」のような常識確認で止めると、OSが入力を情報へ整える効用が見えない。
- 類比できる日常シーン: 注文票、受付、順番待ちの列。
- 採用するシーン比喩: イベントは注文票、イベントループは注文を待ち処理して次を待つ店の流れ。比喩のあと、技術的にはアプリがイベントを順に処理すると戻す。
