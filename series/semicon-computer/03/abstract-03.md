# abstract-03

## ページタイトル

OSとAPI：アプリは窓口を通じてお願いする

## 目標ボリューム

- `draft-03.md` は本文だけで50行程度、約1000字程度を目安にする。
- 図は5点。アプリとOSとハードウェア、OSの管理対象、API窓口、依頼と許可、複数アプリの共存を置く。

## 構成、節見出し、行数目安

### 1. OSは間に立つ: 8-10本文行

- アプリが画面、ファイル、音、カメラを扱う日常表現から入る。
- 一般的なスマホやPCでは、アプリとハードウェアの間にOSがいると説明する。

### 2. OSは端末の資源を管理する: 12-14本文行

- CPU、メモリ、保存場所、画面、入力装置、カメラを管理対象として並べる。
- 複数アプリが同じ端末を使うために、OSの調整が必要と説明する。

### 3. APIはお願いの窓口: 12-14本文行

- アプリの頼みごとを具体的に並べる。
- APIは、アプリがOSや他の機能へ依頼するための決まった入口と説明する。

### 4. システムコールという深い窓口: 8-10本文行

- ファイル読み込み、メモリ確保、別処理開始などOS中核への依頼を入口だけ扱う。
- APIとの関係は詳説せず、OSの深い部分への窓口として位置づける。

### 5. 共存できる理由: 10-12本文行

- OSが順番や範囲を調整するから複数アプリが共存できる。
- 「アプリが保存する」「アプリがカメラを使う」は短縮表現だと回収する。

## 説明すること

- OSはアプリとハードウェアの間に立つ基本ソフトウェアである。
- OSは端末の資源を管理する。
- APIはアプリがOSや機能に依頼するための決まった窓口である。
- システムコールはOSの中核に近い依頼の入口として扱う。

## 中身や仕組みがわかった楽しさ

- 「アプリがカメラを動かす」という言い方が、実はOSを介した短縮表現だと見え直す。
- APIという硬い語が、受付窓口として説明できる語になる。
- 許可画面、対応機種、OSアップデートが、アプリとOSの役割分担につながり始める。

## 構造分析とシーン比喩

- 概念・用語・仕組み: OS、API、システムコール、CPU、メモリ、ファイル、画面、デバイス。
- 理解上昇の入口（既知寄り）: OSがなければ、複数アプリが同じ画面、メモリ、音、保存場所、カメラをそれぞれ取り合い、優先順位や範囲を共通に決めにくくなる。
- 役割: アプリは目的を持つ利用者、OSは資源の管理者、APIは依頼の窓口、ハードウェアは実際に動く設備。
- 情報の流れ: アプリの要求 -> API -> OSの判断 -> ハードウェアやファイル処理 -> 結果がアプリへ戻る。
- 時間順の変化: 依頼する -> OSが確認する -> 資源を使う -> 結果を返す。
- 失敗パターン: APIをインターネット上のサービスだけの語として扱うと、OSの窓口の意味が薄れる。
- 類比できる日常シーン: ビルの管理人、受付窓口、設備利用の申請。
- 採用するシーン比喩: OSはビルの管理人、APIは受付窓口。比喩のあと、技術的にはOSがCPU、メモリ、ファイル、デバイスを管理すると戻す。

## 本文で避けること

- OSをホーム画面や見た目だけとして説明しない。
- アプリが絶対にハードウェアへ直接触れないと断定しすぎない。
- システムコールの詳細へ深入りしない。
